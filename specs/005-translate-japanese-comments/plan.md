# Implementation Plan: Translate Japanese Source-Code Comments to English

**Branch**: `005-translate-japanese-comments` | **Date**: 2026-05-30 | **Spec**: [spec.md](./spec.md)

## Summary

Translate Japanese in-code comments to English across 86 source files (`lib/`, `test/`, one
Android `.kt`) with zero behavioral change. Executed as a multi-agent workflow (one agent per
file, Haiku model) that edits comments only, followed by a mechanical verification gate.

## Technical Context

**Language/Version**: Dart 3.10+ / Flutter 3.38.6-stable; Kotlin (1 Android file)
**Change type**: Comments only — no code, strings, identifiers, or test data
**Testing**: `flutter analyze` + `flutter test` as the regression gate (results must match baseline)
**Scope**: 86 files containing Japanese comments (manifest generated by a Unicode-range grep)
**Out of scope**: `docs/`, `specs/`, README, generated files, user-facing strings

## Constitution Check

| Principle | Compliance |
|-----------|------------|
| I. Type Safety | No type/code changes; comments only. |
| II. KISS & YAGNI | Pure mechanical translation; no abstractions added. |
| III. Test-First | No production logic changes, so no new tests; the *gate* is the existing suite staying green (SC-002), which is the correct verification for a comment-only change. |
| IV. Security-First | No behavioral change; ensures no secrets are introduced into comments. |
| V/VI. SOLID/DRY | N/A (no structural change). |

**Naming**: No identifiers touched. **Result**: PASS.

## Execution strategy

1. **Baseline**: record `flutter analyze` issue count and `flutter test` pass/fail before changes.
2. **Manifest**: list every file with Japanese characters in `lib/**.dart`, `test/**.dart`,
   `android/**.kt` (excluding generated files).
3. **Translate (workflow)**: fan out one Haiku agent per file. Each agent translates only the
   comments in its single file via in-place edits, preserving all code/strings/identifiers/test
   data and formatting. Files are disjoint, so no write conflicts (no worktree isolation needed).
4. **Verify**:
   - Re-grep for Japanese: only intentional test-data strings may remain (SC-001).
   - `flutter analyze` (no new issues) and `flutter test` (no new failures) vs. baseline (SC-002).
   - `dart format` introduces no changes to the touched new files; spot-review a sample (SC-003).
5. **Commit & PR** into the fork's `main`.

## Risks & mitigations

- *Agent edits code by accident* → caught by `flutter analyze` + full `flutter test`; spot review.
- *Agent translates a test-data string (e.g. `あいうえお`)* → instruction forbids touching strings;
  tests asserting those exact values would fail and catch it.
- *Romaji identifiers mistaken for translatable text* → instruction forbids identifier changes.
